# serve.py → TypeScript + Proto-Native Rewrite Plan

**Requested by:** koole__ (2026-03-25)
**Status:** Approved by framers, not yet started
**Language decision:** TypeScript (Bun) with connect-es + buf — chosen over Python connect-python for type safety throughout and because connect-es is the reference connect implementation (more mature)

## Context

serve.py is 1824 lines of handrolled stdlib HTTP (ThreadingTCPServer). It handles routing, parsing, serialization, auth wiring, SSE streaming, OAuth, static files, LLM dispatch, persona CRUD, congress CRUD, task listing, and wallet balance — all in one synchronous file. Features and bugs are hard to fix because of the tangled scope.

## Decision

Rewrite serve.py using:
- **TypeScript** as the implementation language
- **connect-es** (`@connectrpc/connect`) for the proto-native API layer — the reference Connect implementation, more mature than connect-python
- **buf** for code generation from `.proto` files (TypeScript + Python outputs from same proto)
- **Bun** as the runtime (or Node + fastify adapter)
- **Temporal worker stays Python** — LLM SDKs (Anthropic, Grok) are most mature in Python; Temporal Python SDK stays

## What Changes

| Layer | Current | After |
|---|---|---|
| Language | Python | TypeScript |
| Routing | Handrolled do_GET/do_POST/do_PATCH/do_DELETE | Generated by buf from .proto |
| Request parsing | Manual json.loads + field extraction | Generated typed classes |
| Response serialization | Manual json.dumps + send_response | Generated |
| Auth wiring | Manual at each call site (~80 lines) | Middleware |
| SSE streaming | Custom polling loop + _active_streams dict | Server streaming RPC |
| Server | ThreadingTCPServer | Bun.serve (async, no thread-blocking) |
| Temporal activities | Raw aiohttp + manual JSON dicts | Generated typed Python clients (from same proto) |
| Browser client (congress.html) | EventSource SSE | @connectrpc/connect-web streaming client |

## What Stays Python

- Temporal worker (congress_act.py, congress_wf.py, etc.) — LLM SDK interop
- GitHub OAuth flow stays as raw HTTP route alongside ConnectRPC
- GitHub webhook stays as raw HTTP route (HMAC validation, Temporal trigger)

## 5 Proto Services

```
CongressService
  - StartSession(topic, discord_user?) → (session_id, session_number)
  - PostDebate(task, identity, session_id?) → (response, identity)
  - StreamDebate(session_id) → stream of (identity, text, done)     [server streaming, replaces SSE]
  - ListIdentities() → stream of identity objects
  - ListSessions() → stream of session objects
  - GetSession(session_id) → session
  - PatchSession(session_id, verdict?, status?, evolution?, thread_id?, task_titles?) → ok

PersonaService  [localhost-auth only]
  - ListPersonas() → stream of persona objects
  - GetPersona(name) → persona + prompt
  - CreatePersona(name, display_name, model, ...) → persona
  - UpdatePersona(name, fields...) → persona
  - DeletePersona(name) → ok
  - PostVerdict(name, verdict, date?) → ok

AgentService  [public read]
  - ListAgents() → (active[], fired[], moderator)

TaskService
  - ListTasks() → stream of task objects

WalletService
  - GetBalance() → (address, balance_eth, chain)
```

## Chronological Work Breakdown

### Phase 1 — Proto Contracts (1 session)
- Write 5 `.proto` files in `bigclungus-meta/proto/`
- Write `buf.gen.yaml` for TypeScript output (connect-es + @bufbuild/protobuf) AND Python output (for Temporal activities)
- `buf generate` → TS interfaces + service base classes + typed clients; Python typed clients for temporal-workflows
- Commit — single source of truth for all API contracts

### Phase 2 — TypeScript Server Scaffold (1 session)
- Create `clunger/` project with Bun
- Wire connect-es service base classes
- Set up auth middleware (X-Internal-Token + session cookie)
- GitHub OAuth + webhook as raw HTTP routes alongside ConnectRPC
- Static file serving via Bun.serve
- Update `website.service` to run Bun instead of Python

### Phase 3 — Port CongressService (1-2 sessions, largest)
- `StartSession`, `GetSession`, `ListSessions`, `PatchSession` — SQLite (personas.db) + JSON session files
- `ListIdentities` — read persona .md files, parse YAML frontmatter
- `PostDebate` — LLM dispatch (Claude/Grok/Gemini) — port from Python SDK calls to JS SDK equivalents
- `StreamDebate` — server streaming RPC, replaces SSE + `_active_streams` polling

### Phase 4 — Port Remaining Services (1 session)
- PersonaService (CRUD against personas.db + filesystem — .md file mutations)
- AgentService (public agent roster + verdict stats from DB)
- TaskService (read task files from bigclungus-meta/tasks/)
- WalletService (Base RPC call for ETH balance)

### Phase 5 — Update Callers (1 session)
- `congress.html` SSE client → `@connectrpc/connect-web` streaming client
- Temporal Python activities → use generated Python connect client (replaces raw aiohttp calls to serve.py)

### Phase 6 — Cutover + Cleanup (1 session)
- Run both servers in parallel, verify all congress sessions work end-to-end
- Delete serve.py
- Update Cloudflare tunnel if port changes

**Total estimated: 5-7 sessions. Temporal worker stays Python throughout.**

## Parallel Development Approach

Per koole__ (2026-03-25): build `clunger/` **concurrently** with serve.py, not as an inline replacement. This allows behavioral comparison throughout development.

**Setup:**
- `clunger/` runs on port 8081
- `website.service` keeps serving serve.py on :8080 throughout development
- New `clunger.service` systemd unit for the TypeScript server
- Both services run simultaneously; no cutover until `clunger` is verified

**Verification approach:**
- Hit the same endpoint on both servers, diff the responses
- Flag behavioral differences as bugs or intentional improvements before cutover
- Cutover = swap which port Cloudflare tunnel routes to (one-line config change)

**Rollback:**
- If `clunger` has issues after cutover, flip the Cloudflare route back to :8080
- serve.py stays intact until both framers confirm `clunger` is stable
